Platform vs Application

The boundary between generic platform code and mower-specific application code. The discipline document: what goes where, what the forbidden words are, and how reviews enforce it.

1.0

Purpose

This platform is named platform-first and built application-first. The intended arc is that the lawn mower is the first application; the next might be a pool-cleaning variant, a snowplow, a dog-walk companion. For that arc to pay off, there has to be an enforced discipline about what kind of code is allowed to reference "mowing."

This document is that discipline. It is the most prescriptive document in the corpus. When in doubt about where something belongs, reread §2 and §4.

Why this matters

Every commercial robotics project either maintains this discipline from day one and then pivots smoothly when the second application arrives, or lets the discipline slip and spends the second application's first year doing an ugly extract. We are doing this from day one.

2.0

The Rule

Fundamental rule

Nothing in /platform/ references lawn mowing, grass, blades, or any other mower-specific concept. Any code that does belongs in /applications/mower/.

This is a simple, testable, lint-enforceable rule. Violations are caught by (a) code review, (b) a CI lint that rejects the forbidden vocabulary below in platform/**, and (c) the corpus itself — no platform_* ICD or SRS names a mower-specific concept.

3.0

Vocabulary Discipline

When writing platform code or platform documents, use the generic term. The mower-specific term is reserved for /applications/mower/.

Application term (mower)Generic platform termRationale
bladetoolA blade is one of many possible end-effectors. The platform has a generic "tool" concept with engage/disengage semantics.
grassterrain · surfaceGrass is one of many surfaces. Sensing and traction models are surface-generic.
yardarea · zoneA yard is one instance of an area. Zones are polygons; the platform doesn't know what's inside them.
lawnzoneSame reason as yard.
mowcover · traverseCoverage planning is a platform concern; mowing is application-specific.
mowing sessionmissionThe platform has missions. Mowing missions are a mower-layer concept.
cut heighttool.intensityPlatform has a 0.0–1.0 tool-intensity field; mower maps that to cut height.
mulch / collect / bagtool_modeApplication-defined enum. The platform does not know these modes.
The lint

A CI lint checks platform/**/*.{py,cpp,hpp,msg,srv,action,yaml,md,html} for the regex (blade|grass|yard|lawn|mow|mulch) (case-insensitive, word boundaries). Matches fail CI. Overrides require an inline PLATFORM_LINT_OK: <reason> comment that is reviewed as part of code review.

4.0

What Goes Where

4.1 Platform packages
  • platform_hal — motor, IMU, e-stop, camera drivers. Vendor-specific internally; vendor-neutral at the ROS surface.
  • platform_msgs — all generic message types (SafetyState, ToolCommand, ZoneBoundary, AerialImageMeta).
  • platform_localization — wheel/IMU fusion, SLAM integration, EKF configuration.
  • platform_perception — YOLO, detection filtering, costmap producers.
  • platform_planning — coverage planner (iter 2+), waypoint navigation.
  • platform_safety — safety_monitor, geofence_enforcer.
  • platform_mapping — drone mission coordinator, image retrieval, georeferencing, boundary ingestion.
  • platform_api — FastAPI mission service, telemetry WebSocket.
4.2 Application packages (mower)
  • applications/mower/mower_app — mower-specific mission logic: start mowing, pause mowing, "always mow in straight lines," etc.
  • applications/mower/mower_launch — launch files that compose the platform nodes with mower-specific parameter files.
  • applications/mower/mower_urdf — URDF describing the mower chassis (iter 3+).
  • applications/mower/mower_config — mower-tuned parameters: tool-type enum, tilt limits for blade operation, blade RPM targets.
  • applications/mower/mower_calibration — blade-specific calibration procedures (iter 2+).
  • applications/mower/tool_controller — future blade-on/off and cut-height logic. Subscribes to /hal/tool_cmd_safe; drives the physical blade motor.
4.3 Ambiguous cases, resolved
QuestionAnswer
Where do mower-specific tilt limits live?Platform accepts a tilt_limit_deg parameter. The mower config overrides the default. Platform code does not know that "blade needs tighter limits."
Where does "no mowing while it's raining" live?Application. The platform has no weather sensor (see system context). A future rain-detection node would be application-level unless weather becomes a platform-wide concern.
Where does "sharpen blade every N hours" live?Application. The platform has a tool_hours diagnostics field; the mower interprets and notifies.
Where does "slow down on dewy grass" live?Application, when it exists. The platform has velocity scaling based on terrain surface classification (iteration 3+); the mower maps "dewy" to "lower traction."
Where does "pose must be inside zone" live?Platform. Geofencing is not mower-specific. The platform enforces it; applications set the active zone.
5.0

Message-Level Discipline

Message types are part of the contract surface. A message named BladeCommand would taint every node that touches it. Two rules:

  1. Platform messages (in platform_msgs) are generic. ToolCommand, not BladeCommand. ZoneBoundary, not YardBoundary.
  2. Application messages (in mower_msgs, when needed) may carry application vocabulary, but should be rare. The strong preference is to add fields to platform messages with generic names that applications interpret.

Example: the ToolCommand message has a tool_type field of type uint8. The values of that field are application-defined. The platform doesn't know that tool_type == 1 means "blade" — that mapping lives in mower_config.

Defaults are fine; values are not names

Platform code may hard-code a parameter default named max_linear_vel: 0.7. It may not hard-code a parameter default named mower_max_vel. Names leak intent; values don't.

6.0

Configuration & Parameters

Parameter precedence (highest wins):

  1. Runtime overrides (command-line, environment, operator app setting).
  2. Application config (mower_config/config/*.yaml).
  3. Platform default (declared in the node's C++/Python source).

Platform defaults are reasonable generic values, not tuned for any application. The mower application provides tuned values as overrides. A pool-cleaner application in the future would provide a different set.

Both platform defaults and application overrides are documented in the relevant SRS under the "Configuration Parameters" section.

7.0

Review Checklist

For any PR touching platform/**, the reviewer asks:

  • Does any identifier or comment contain a forbidden word (blade, grass, yard, lawn, mow, mulch)?
  • If the change introduces a new parameter, is the name generic?
  • If the change adds a branch on tool type, is the branch in the mower layer rather than the platform?
  • If the change is to a platform message, is the field name generic?
  • Does the change preserve the dependency direction from ARCH-002 §9 (no upward dependency on application code)?

For any PR touching applications/mower/**:

  • Could this logic be generalized and moved into the platform? If yes, move it and replace with a config file.
  • Is the mower layer reimplementing something the platform already offers?
8.0

Open Issues

  1. OPEN-01 — The name "tool" in the platform is slightly awkward for a "tool-less" application (e.g., mapping-only drone). Consider "effector" or "payload." Decision deferred until the second application arrives.
  2. OPEN-02 — Documentation lint. This corpus itself should be lint-enforced for the platform/application distinction. A simple grep in a CI job is sufficient; hold until the corpus is stable.
  3. OPEN-03 — Some safety thresholds are "platform-reasonable but mower-loose" (e.g., tilt limits). Should the platform expose a "safety profile" abstraction rather than individual parameters? Evaluate in iteration 2.
