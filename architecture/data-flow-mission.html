Mission Data Flow

Phase-by-phase walkthrough of a complete iteration-1 mission: the operator taps "Map My Yard," the drone flies, the user draws a boundary, and the rover is teleoperated within that boundary — with safety gating throughout.

1.0

Purpose

This document traces the complete flow of a mission from the operator's first tap to the final rosbag flush. It is the single best way to see how the layered architecture composes in practice, and it is the reference against which interface specifications are sanity-checked.

Each phase below names the participating nodes, the interfaces they use (every interface is a link into the ICD catalog), and the phase's pass/fail signal. Where an interface does not yet exist as an ICD — because iteration 1 stubs the behavior — it is called out explicitly.

2.0

Mission Overview

The iteration-1 mission has six phases, labeled A through F. They run sequentially with the exception of Layer 0 safety, which is always on.

A ─ Intent capture user opens app, taps "Map My Yard" mobile app → mission_api (REST) → drone_mission_coordinator ▼ B ─ Drone mapping arm → takeoff → climb → capture → descent → land drone_mission_coordinator → mavros → FC · XIAO captures image ▼ C ─ Boundary draw aerial image delivered to app; user draws polygon image_retriever · georeferencer → mission_api → app ▼ D ─ Zone activation polygon POSTed; boundary_ingester persists and publishes boundary_ingester → /mapping/current_zone → geofence_enforcer ▼ E ─ Teleop with safety user drives; safety gates every command joy_teleop → twist_mux → safety_monitor → roboclaw_driver geofence_enforcer · yolo_detector · estop_monitor → safety_monitor ▼ F ─ Shutdown & audit operator parks rover, closes app; rosbag flushes rosbag2 → disk · system_monitor final report

The rest of this document walks each phase in detail.

A

Intent Capture

Trigger. The operator opens the mobile app, authenticates if required, and taps the "Map My Yard" button.

Flow.

  1. App issues POST /missions/map with an empty body plus the bearer token. See ICD-API-001.
  2. The mission_api service validates the token and the current system state. Rejects if:
    • Safety state is not OK (e.g., e-stop pressed, low battery) — see /safety/state.
    • A prior mission is still in flight.
  3. mission_api invokes the /mapping/run_mission action on drone_mission_coordinator. Action returns a mission_id immediately.
  4. mission_api returns 202 Accepted with {mission_id, status: PENDING} to the app.
  5. App transitions to the "mapping in progress" screen and begins polling GET /missions/{id}/status at 1 Hz.

Pass signal. The app receives a mission_id. Failure modes: 401 (unauth), 409 (system busy), 503 (safety not OK).

B

Drone Mapping

Span. Typically ~3 minutes from action accept to drone on the ground with image captured.

Sequence.

  1. drone_mission_coordinator enters ARMED state. Sends MAV_CMD_COMPONENT_ARM_DISARM via mavros.
  2. Transitions to TAKEOFF. Commands MAV_CMD_NAV_TAKEOFF with target altitude 30 m.
  3. Transitions to CLIMBING. Waits for altitude ≥ target − 1 m or 45 s timeout.
  4. Transitions to CAPTURING. Commands the XIAO (via xiao_bridge firmware) to capture a still image. Waits for confirmation.
  5. Transitions to DESCENDING. Commands MAV_CMD_NAV_LAND.
  6. Transitions to LANDED. Publishes action feedback to mission_api.
  7. image_retriever polls the XIAO over Wi-Fi; downloads the image to the Pi.
  8. georeferencer reads the image + capture pose; computes corner lat/lons; publishes /mapping/latest_aerial_image.
Safety veto at any point

At any step in this sequence, if the operator presses the physical or soft e-stop, drone_mission_coordinator transitions to ABORTING, commands immediate land, and reports failure. See state-machine detail in SM-DRN-001.

Pass signal. /mapping/latest_aerial_image published with non-empty image_path. Mission state observable as COMPLETE via GET /missions/{id}/status.

C

Boundary Definition

Flow.

  1. App receives COMPLETE; issues GET /missions/{id}/result.
  2. mission_api responds with the AerialImageMeta payload + a binary handle to the image (file download endpoint).
  3. App displays the georeferenced image with map controls.
  4. Operator draws the outer boundary polygon by tapping vertices. Optionally adds exclusion polygons (tree wells, flower beds).
  5. Operator taps "Save Zone"; app issues POST /zones with the ZoneBoundary payload and a zone name.

Pass signal. 201 Created with saved zone metadata.

Note on validation. The app enforces basic topological checks (self-intersection, minimum area) client-side. The server revalidates server-side because we don't trust clients; see boundary_ingester for the server-side validation ruleset.

D

Zone Activation

Flow.

  1. Operator selects the newly saved zone (or any previously saved zone) in the app and taps "Activate for mowing."
  2. App issues POST /zones/{name}/activate (or equivalent; see ICD-API-001).
  3. mission_api calls the /mapping/load_zone service on boundary_ingester.
  4. boundary_ingester publishes the activated /mapping/current_zone as a latched topic.
  5. geofence_enforcer receives the zone; begins comparing fused pose to the polygon at 10 Hz; publishes /geofence/state.
  6. safety_monitor consumes /geofence/state; begins treating OUTSIDE as an ESTOP trigger.

Pass signal. /mapping/current_zone carries the requested polygon; /geofence/state transitions from INSIDE with distance_to_boundary_m matching the rover's actual position.

E

Teleop with Safety

The steady-state operational mode for iteration 1. Operator drives the rover manually within the activated zone; all safety subsystems are engaged.

Command path (one command cycle, ~20 ms):

t=0 user touch input on virtual joystick t+5ms app serializes Twist to JSON t+8ms REST POST /rover/teleop or equivalent real-time channel (WS or UDP — open) t+12ms mission_api rclpy-publishes to /hal/cmd_vel_teleop t+13ms twist_mux republishes (if this is the highest-priority source) on /hal/cmd_vel_raw t+14ms safety_monitor evaluates safety state: OK → passthrough to /hal/cmd_vel_safe not OK → publishes zero (with decel ramp) to /hal/cmd_vel_safe t+16ms roboclaw_driver converts to per-track velocities; issues serial command t+20ms motors reflect commanded velocity

Parallel safety evaluation (continuous):

/perception/people 10 Hz ─┐ /perception/obstacles 10 Hz ─┤ /hal/imu/data 100 Hz ─┤ /hal/estop_pressed 20 Hz ─┼─► safety_monitor ─► /safety/state ─► app, rosbag /geofence/state 10 Hz ─┤ ─► /hal/cmd_vel_safe (gate) /hal/battery 1 Hz ─┘

Pass signal. The rover moves in response to operator inputs; stops immediately on any safety trigger; telemetry reflects all of this to the operator in < 1 s.

F

Shutdown & Audit

  1. Operator parks rover in a safe spot; releases the virtual joystick.
  2. Operator presses the physical e-stop or issues POST /rover/estop.
  3. safety_monitor enters ESTOP; motors fully stopped.
  4. Operator closes the app; WebSocket telemetry disconnects cleanly.
  5. rosbag2, which has been recording since system startup, is flushed on graceful shutdown or at 500 MB rollover.
  6. system_monitor emits a final summary on /diagnostics: total runtime, peak CPU/memory, transitions logged, any unresolved warnings.

Pass signal. Rover at rest with motors inert; rosbag file closed; no unexpected error-level diagnostics in the final report.

4.0

Open Issues

  1. OPEN-01 — Teleop transport: REST per-command is inefficient for continuous joystick input. Evaluate UDP or a dedicated WebSocket stream (contract in ICD-API-002 reserved for telemetry only at present).
  2. OPEN-02 — Autonomous return-to-dock (Phase F.5) is planned for iteration 2. Iteration 1 requires operator to manually park.
  3. OPEN-03 — Multiple zones per home: activation is one-at-a-time in iteration 1. Iteration 2 may introduce automatic scheduling across zones.
