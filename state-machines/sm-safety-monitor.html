Purpose

Formal specification of the four-state machine implemented inside safety_monitor. Every transition, every invariant, every timing property of that node's core loop is captured here.

2.0

Diagram

Startup: always enters ESTOP with reason "startup — waiting for topics". Transitions to OK only after every subscribed topic has been received at least once.

Invariants

  • INV-1. tool_permitted ⇒ state ∈ {OK, WARNING}. Tool is never permitted in ESTOP or CRITICAL.
  • INV-2. cmd_vel_safe = 0 when state = ESTOP, within the decel-ramp tolerance. Immediate zero if tilt-triggered.
  • INV-3. /safety/state published at ≥ 10 Hz under all conditions. No silence.
  • INV-4. Hardware heartbeat (iteration 2+) pulses at 100 Hz with < 5 ms jitter. Iteration 1: placeholder only.
  • INV-5. Latched ESTOPs require /safety/reset. Auto-clearable do not.
  • INV-6. /safety/reset returns success only if all underlying trigger conditions are currently clear.
  • INV-7. Stale topic data is treated as worst case. Absence of data is unsafe.
