Mission REST API

HTTPS REST surface between the operator's mobile application and the rover. Served by mission_api on the Pi.

1.0

Purpose

Defines the HTTPS REST surface the operator app uses to command and observe the rover. Every endpoint is authenticated via bearer token and served by the mission_api FastAPI service. This is the external contract; internal rclpy calls from the service to ROS 2 nodes are not part of this ICD.

4.0

Authentication

Every endpoint requires a bearer token:

Authorization: Bearer <token>

Tokens are issued manually via a CLI tool (rover-token issue --name "iPhone") and stored hashed in /var/lib/platform/tokens.db. Revocation is by CLI (rover-token revoke <id>). Iteration 1 has no self-service enrollment; the operator pairs devices manually.

TLS is mandatory. Self-signed certificate pinned in the mobile app at install.

5.0

Error Model

HTTPMeaning
200 / 201 / 202 / 204Success (RESTful conventions)
400Malformed request (e.g., self-intersecting polygon)
401Missing or invalid bearer token
403Token valid but action not permitted
404Mission id / zone name not found
409Resource conflict (e.g., mission already running)
503Safety state disallows action (e.g., teleop while ESTOP)
500Unexpected internal error

Error responses carry a JSON body: {"error": "...", "detail": "..."}.

Timing & Rate

EndpointTarget latency (p95)
/rover/state≤ 50 ms

/missions/map≤ 500 ms accept

No explicit rate-limiting in iteration 1; /rover/teleop is expected to be called 20–30 Hz during active joystick use.

7.0

Fault

  • mission_api crash: systemd restart. Connections drop; client reconnects; operator sees transient "disconnected" state in app.
  • ROS 2 side unavailable (safety_monitor down): Endpoints returning rover state return 503 with message.
  • Invalid token: 401.
8.0

Requirements

  • REQ-ICD-API-001-01
    All endpoints shall require a valid bearer token; requests without one shall be rejected with 401 before any side effect.
    verified by: TEST-API-001
  • REQ-ICD-API-001-02
    /rover/estop shall cause /safety/state to transition to ESTOP within 200 ms of the HTTP request returning 200.
    verified by: TEST-API-002
  • REQ-ICD-API-001-03
    Teleop commands received when safety state is ESTOP shall be ignored and return 503.
    verified by: TEST-API-003
