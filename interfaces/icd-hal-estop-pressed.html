1.0

Purpose

The physical e-stop button is a normally-closed (NC) bulkhead switch wired in series with the motor power rail. Pressing it cuts motor power with no software involvement. It is also wired to a GPIO input pin on the Pi so that software can observe the state.

This topic carries that observation. It is safety-critical because safety_monitor uses it to propagate the e-stop state into /safety/state, which downstream systems (mobile app, mission_api, logs) consume.

Independence principle

The hardware e-stop path does not depend on this topic. Motors stop when the button is pressed because power is cut — not because this topic tells them to. This topic informs the software; the hardware is the enforcement.

3.0

Message

Type: std_msgs/msg/Bool. Field data: TRUE = button pressed (motor power cut); FALSE = button released.

4.0

Semantics

NC switch semantics: the wire is closed (GPIO reads LOW via pull-up to GND) when the button is released. Pressing opens the circuit; GPIO reads HIGH. The driver inverts so that data = true means pressed.

Debouncing: GPIO is sampled at 50 Hz; state changes are accepted only after 3 consecutive matching reads (60 ms debounce).

Transitions are published on-change (within one sample period) AND periodically at 20 Hz as a liveness keepalive. The safety monitor uses the keepalive to detect failure of the estop_monitor node itself.

Fault

Broken wire (NC fails open)GPIO reads HIGH permanently → reported as pressed → ESTOP. Correct fail-safe.
Stuck-closed wire (NC stuck pressed)Button cannot register release. System stays in ESTOP until physical repair. Detected by startup self-test (§8).
