Telemetry WebSocket

Persistent WebSocket stream pushing live telemetry from the rover to the operator's mobile app.

Document ID

1.0

Purpose

A single push channel from the rover to the mobile app. The app connects once at startup; the rover streams pose, safety state, battery, detection counts, and mission state at 5 Hz. Replaces per-second polling on /rover/state.

2.0

Participants

Server
telemetry_publisher
SRSSRS-API-002Pathwss://rover.local:8443/ws/telemetry
Client
Operator mobile app
AuthBearer token passed in Sec-WebSocket-Protocol header; validated at handshake.
3.0

Connection

Single WebSocket. On successful handshake, server sends an initial snapshot message, then periodic updates. Client may send ping control frames; server responds with pong. No other client-to-server messages are defined in iteration 1.

Multiple client connections are permitted (e.g., a phone and a tablet). Each connection independently receives the same stream.

4.0

Payload

Server messages are JSON text frames:

{ "timestamp": 1745084320.247, "pose": {"x": 12.34, "y": 5.67, "yaw_rad": 1.234, "covariance_trace": 0.08}, "safety_state": { "state": "OK", "reasons": [], "motion_permitted": true, "tool_permitted": true }, "battery": {"percentage": 0.82, "voltage": 15.4}, "detection_count": {"people": 0, "obstacles": 2}, "mission_state": "IDLE", "geofence": {"state": "INSIDE", "distance_m": 3.2} }

Fields MAY be extended (additive) in future revisions; clients must tolerate unknown keys. Removal or type change requires a new document revision.

5.0

Timing

5 Hz nominal. Burst updates (on safety state change) push an additional message immediately so operator feedback is not limited by cadence.

6.0

QoS / Transport

TCP-based. Latency-sensitive but not loss-sensitive at the transport layer (TCP handles). If the connection is idle > 30 s, server sends a ping; if client does not respond within 10 s, connection is closed and client is expected to reconnect.

7.0

Fault

FaultResponse
Network disconnectClient reconnects with exponential backoff (1 s → 2 s → 4 s → max 30 s). Server accepts.
telemetry_publisher crashConnection closed by server-side termination; systemd restart; client reconnects.
Upstream ROS 2 topics silentServer continues publishing with stale fields; flags staleness in payload (TBD field).
Invalid token at handshakeHTTP 401 before WebSocket upgrade.
8.0

Requirements

  • REQ-ICD-API-002-01
    Messages shall be pushed at 5 Hz ± 1 Hz during a healthy connection.
    verified by: TEST-API-010
  • REQ-ICD-API-002-02
    Safety-state changes shall trigger an immediate burst message (out-of-cadence) within 100 ms of the change.
    verified by: TEST-API-011
  • REQ-ICD-API-002-03
    Handshake shall fail with 401 when an invalid bearer token is provided.
    verified by: TEST-API-012
