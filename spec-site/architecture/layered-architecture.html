ARCHITECTURE · STRUCTURE

Layered Architecture

Five functional layers plus a cross-cutting safety layer. Each layer owns a concern, exposes its capabilities through a defined interface, and is permitted to depend only on strictly lower layers.

1.0

Purpose

This document defines the five functional layers of the platform and the cross-cutting safety layer. It is the reference all node SRSes implicitly obey — each node has a primary home layer, each interface crosses zero or one layer boundaries, and the permitted dependency direction is strictly downward (plus Layer 0).

The pattern is standard: isolate hardware in a vendor-specific HAL, build abstract capabilities above it, compose them into mission logic at the top. What distinguishes this system is the independent cross-cutting safety layer, which reads from every relevant layer and has veto authority over actuator outputs.

Sibling docs

For the physical allocation of layers to hardware, see Deployment. For the boundary between platform code and application code (orthogonal to layering), see Platform vs Application.

Layer 0 is drawn at the bottom by convention but it reads from and writes to every layer. It is not a "stage" data flows through — it is a constant presence that can interrupt at any point.

3.0

Layer 0 · Safety (cross-cutting)

Concern. Ensure no actuator commanded motion occurs when safety conditions are violated. Operate independently of all higher layers so that a bug in mission, autonomy, or even perception cannot defeat the safety constraint.

Composition.

  • Physical e-stop. NC bulkhead switch in series with the motor power path. Pressing it cuts motor power without software involvement. Iteration 1 implementation.
  • Safety MCU. Independent microcontroller with its own firmware. Iteration 2+ when a blade tool is introduced. Enforces heartbeat, tilt, and blade-on-lift interlocks in hardware.
  • ROS safety monitor. The software-layer gate. Aggregates perception, sensor, and hardware inputs into a unified /safety/state and controls the /hal/cmd_vel_safe gate. See SRS-SAF-001.
  • Geofence enforcer. Compares fused pose with active zone polygon. Publishes /geofence/state that the safety monitor consumes.
Independence principle

The three safety layers (hardware e-stop, safety MCU, ROS safety monitor) are designed to fail independently. Loss of any one layer degrades safety but does not defeat it. This is the single most important invariant in the architecture.

4.0

Layer 1 · HAL

Concern. Translate between vendor-specific hardware protocols and vendor-neutral platform message types. Above this layer, a swap of motor controller, IMU, or flight controller affects no code — only the driver.

Rule. A HAL node may only publish platform-generic message types. A HAL node is permitted to read vendor-specific registers and speak vendor-specific protocols, but must not leak those details into its ROS topics.

5.0

Layer 2 · Control

Concern. Turn a desired velocity (from the operator, from the autonomy layer, or from a teleop stick) into motor-level commands. Arbitrate between competing velocity sources.

Components:

  • twist_mux. Off-the-shelf node. Accepts multiple Twist inputs (teleop, autonomous, recovery) with priorities. Publishes the highest-priority currently-active source on /hal/cmd_vel_raw.
  • skid-steer kinematics. Lives inside roboclaw_driver and converts Twist to per-track velocities. Not a separate node in iteration 1; may become one in iteration 3 if the rover chassis changes.
Rationale: twist_mux instead of custom arbitration

twist_mux is well-tested, timeout-aware, and ROS-native. Writing a custom priority multiplexer would be strictly worse at day one and no better at year three. The one thing it doesn't do is safety gating — which is why the pipeline is twist_mux → safety_monitor → motor driver, not twist_mux → motor driver.

6.0

Layer 3 · Perception, Localization, Mapping

Concern. Build the world model the rover reasons against. Three sub-concerns:

  • Perception. What objects are where. YOLO on OAK-D Lite produces detections; a filter splits them into living things (safety input) and generic obstacles (navigation input in iter 2+).
  • Localization. Where is the rover. Local pose from wheel odometry + IMU fused via EKF; global pose from local + SLAM (RTAB-Map) + optional GPS fused via a second EKF.
  • Mapping. What is the yard. Drone-captured aerial image georeferenced into world coordinates; user draws boundary polygons on it; the polygon becomes the active geofence zone.

The perception and localization sub-layers are operationally independent of each other (an EKF does not need detections; YOLO does not need pose in iteration 1), but both feed upward into autonomy and safety.

7.0

Layer 4 · Autonomy

Concern. Higher-order decisions: "go to this waypoint," "cover this area," "return to dock," "execute a mapping mission." This layer is deliberately small in iteration 1.

  • drone_mission_coordinator. The one substantial iteration-1 autonomy component. Orchestrates arm → takeoff → climb → capture → descent → land via MAVROS. Exposed as a ROS action.
  • Waypoint navigator. Stubbed in iteration 1 (manual teleop). Full Nav2 stack in iteration 2+.
  • Coverage planner. Not implemented until iteration 2.
  • Behavior tree. Not implemented until iteration 2.
8.0

Layer 5 · Mission & UI

Concern. The operator-facing surface. Authorization, presentation, and the translation of human intent into layer-4 actions.

  • Mobile app. Presents the aerial image, captures boundary polygons, offers teleop controls, displays live telemetry. Built as a client in React Native or Flutter (decision open).
  • mission_api. FastAPI service on the Pi. Holds all REST endpoints; bridges to ROS 2 via rclpy. See ICD-API-001.
  • telemetry_publisher. WebSocket publisher at 5 Hz. See ICD-API-002.
  • boundary_ingester. Persists drawn polygons to disk; serves them via service and publishes the active zone.

This layer is the only one the operator's device touches. Everything below it runs on the Pi or on the drone.

Crossings & Rules

Permitted dependency directions.

FromMay depend onMay not depend on
Layer 5Layer 4, Layer 3, Layer 0Layer 1 directly (must go through 2/3)
Layer 4Layer 3, Layer 2, Layer 0Layer 5, Layer 1 directly
Layer 3Layer 1, Layer 0Layer 2, 4, 5
Layer 2Layer 1, Layer 0Layer 3, 4, 5
Layer 1Layer 0Layer 2, 3, 4, 5
Layer 0Reads from all; writes only to Layer 1 (via veto topic) and Layer 5 (via /safety/state)—

Violations that are rejected in review:

  • A layer-5 node directly subscribing to /hal/imu/data (skip) — must go through an EKF or dedicated aggregator.
  • A layer-3 node writing to /hal/cmd_vel_* (skip + downward write) — detection nodes must not drive motors.
  • A layer-2 node including YOLO output in its kinematics (upward dependency) — control logic must not depend on perception.
  • Bypassing the safety gate with a direct publication to motor commands.
The one exception: diagnostics

Any node may publish to /diagnostics. Any node may subscribe to /diagnostics for observability. This is explicit cross-cutting concern handling and is not considered a layer violation.

Open Issues

  1. OPEN-01 — Should skid-steer kinematics be a separate node for testability, or is inlining it in roboclaw_driver acceptable? Current plan: inline for iteration 1, split if iteration 3 chassis work demands it.
  2. OPEN-02 — Layer-5 endpoints on the Pi bridging to ROS 2 via rclpy are a single process today. At scale, should REST and telemetry be split into separate processes for isolation? Deferred to post-M6.
  3. OPEN-03 — The living_thing_filter sits in Layer 3 but is safety-critical. Should safety-critical Layer 3 nodes be mirrored under Layer 0's governance? Currently: no, they emit to Layer 0 topics but run as Layer 3 nodes.
