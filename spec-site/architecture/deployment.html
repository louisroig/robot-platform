Purpose

The layered architecture document described the platform's functional stack. This document describes its physical realization: which chips run which code, how they talk, and how they are powered.

Allocation decisions that matter for code authors (callback groups, QoS, latency budgets) are noted inline and cross-referenced to the relevant ICDs.

3.0

Ground Unit (Rover)

The rover's compute is a single Raspberry Pi 5 running Ubuntu Server 24.04 and ROS 2 Jazzy. Every custom ROS node in the corpus — unless marked "drone side" — runs here.

3.1 Pi 5 software stack
  • OS: Ubuntu Server 24.04 LTS (arm64).
  • ROS 2: Jazzy Jalisco. DDS implementation: default Cyclone DDS.
  • Python: 3.12 (for the rclpy-based nodes and mission_api).
  • C++: gcc 13 (used by safety_monitor).
  • Service supervision: systemd units per node group (hal, perception, safety, mission, telemetry).

GPIO pin stability

GPIO pins for PWM and DIR are configured via the motor_driver node parameters. Ensure pinout diagram is documented in deploy/ and that pin assignments do not conflict with other HAL peripherals (IMU, e-stop GPIO). Pi 5 GPIO numbering is BCM; verify against the hardware pinout.

4.0

Aerial Unit (Drone)

The drone has two compute nodes: the flight controller (STM32 running ArduCopter) and the XIAO ESP32-S3 bridge.

4.1 Flight controller
  • iFlight Blitz Whoop F7 AIO (STM32F745 MCU, ICM42688 gyro, BlueJay ESC firmware).
  • Flashed with ArduCopter 4.5 (quad-X configuration).
  • Parameters captured as FW-ACP — dump + rationale.
  • TELEM1 reserved for the ELRS receiver's serial telemetry (not used in iteration 1).
  • TELEM2 connected to XIAO ESP32-S3 at 57600 baud for MAVLink.
4.2 XIAO ESP32-S3 Sense
  • Custom firmware — see FW-XIAO.
  • Role 1: Wi-Fi MAVLink bridge. Forwards MAVLink between FC UART and ground station TCP. The Pi's mavros node connects to tcp://xiao.local:5760.
  • Role 2: OV2640 capture. Receives a capture command via a MAVLink message or a side-channel HTTP endpoint; captures a JPEG still; stores on its micro-SD or transfers via Wi-Fi.
  • Powered from drone's 5V BEC rail (shared with RC receiver).
Why not on the FC itself

The Blitz F7 AIO does not have enough GPIO or camera bandwidth for onboard capture, and its firmware is already full-utilization for flight control. The XIAO is a low-cost, low-weight add-on that handles the two things the FC cannot: Wi-Fi and camera.

5.0

Mobile Device

  • Operator's personal iOS or Android phone.
  • Runs the mobile app (framework under evaluation: React Native vs. Flutter — see ARCH-006).
  • Communicates with the Pi via Wi-Fi: REST (ICD-API-001) and WebSocket telemetry (ICD-API-002).
  • No direct ROS 2 connection. No DDS on the phone.

Power & Wiring

6.1 Rover power tree
4S pack (14.8V nominal, ~5Ah) │ ├──► Main power switch (manual rotary) │ │ │ ├──► E-stop NC relay ──► IBT-2 H-bridge motor power (14.8V) │ │ │ │ │ ├──► Left motor │ │ └──► Right motor │ │ │ └──► Buck converter 14.8V → 5V / 4A ──► Pi 5 │ │ │ ├──► OAK-D Lite (USB3 5V 900 mA) │ └──► IMU (Qwiic 3.3V) │ └──► Battery sense tap ──► (iter-1 deferred; future ADS1115 ADC → /hal/battery)
Hard safety path

The e-stop NC relay is in series with the motor power rail, not the logic rail. Pressing it removes motor power while leaving the Pi and all sensing alive — so the system can report the e-stop state, enter the ESTOP software state, and resume without a boot cycle when released.

6.2 Drone power tree
3S 550 mAh pack │ ├──► FC VBAT ──► 5V BEC on FC ──► XIAO ESP32-S3 Sense │ └──► ELRS receiver │ └──► ESC rail ──► 4 motors

The drone's safety story differs from the rover's: in iteration 1 the drone is single-failure tolerant by virtue of being sub-250 g, operator-supervised, and airborne for < 5 minutes per mission. The rover operates around living things and for hours; the drone does not.

Networking

All interdevice communication is over 2.4 / 5 GHz Wi-Fi on the operator's home network.

FlowTransportPort / PathRate
App → mission_apiHTTPSTCP 8443request/response
mission_api → appWebSocket (wss)TCP 8443 / /ws/telemetry5 Hz
Pi → XIAO (MAVLink)TCP (forwarded by XIAO firmware)TCP 5760event-driven
Pi → XIAO (image pull)HTTPTCP 80 on XIAOon-demand
ROS 2 DDSUDP multicast/unicastdefault Cyclone portsper-topic
Operator ELRS RC → droneSub-GHz RFseparate link250 Hz

The Pi is configured to advertise the ROS 2 domain only on localhost (ROS_DOMAIN_ID isolated, ROS_LOCALHOST_ONLY=1) so that DDS traffic does not leak onto the home network. Anything the operator needs goes through the mission_api surface.
