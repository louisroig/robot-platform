1.0

Purpose

This document establishes the boundary of the ground-air autonomous platform as a single system under development. It enumerates the external actors that cross that boundary, the nature of each interaction, and the environmental and regulatory conditions under which the system must operate. It explicitly identifies capabilities and responsibilities that fall outside the platform's scope.

All subsequent architecture documents assume this boundary. When a requirement references "the system," it refers to everything inside the boundary drawn here. When a requirement references an actor, it refers to one of the actors enumerated in §3.

Reading notes

For deeper structure inside the boundary, see Layered Architecture. For the flow of information during a mission, see Mission Data Flow. For physical allocation of the system across hardware, see Deployment.

2.0

System Boundary

The ground-air autonomous platform comprises:

  • The ground unit — a tracked rover carrying a Raspberry Pi 5, a stereo camera with onboard inference, an IMU, a physical e-stop, and a battery.
  • The aerial unit — a sub-250 g mapping drone carrying an ArduCopter-flashed flight controller, a Wi-Fi MAVLink bridge, and a payload camera.
  • The operator mobile application — the human interface for boundary drawing, teleoperation, mission triggering, and live telemetry. Runs on the operator's personal device.
  • All software nodes running on the ground unit (ROS 2 Jazzy), the drone (ArduCopter + XIAO firmware), and the mobile device (client app).
  • All communications linking the above — Wi-Fi between the operator and the rover, Wi-Fi between the XIAO bridge and the ground station, MAVLink over UART between XIAO and flight controller, 4× GPIO hardware PWM (RPWM/LPWM per track) between Pi and the IBT-2 H-bridges, and I²C between Pi and IMU.

Not part of the platform:

  • The residential yard environment itself — terrain, grass, trees, fixtures, weather.
  • The operator's personal device hardware, its operating system, its network provider.
  • The Wi-Fi access point and home network the operator's device and the rover attach to.
  • Any cloud backend. Iteration 1 is strictly local — no data leaves the home network.
  • The charging infrastructure for the rover battery in iteration 1 (the operator plugs it in manually).
  • The FAA LAANC or equivalent airspace authorization system. The operator is responsible for ensuring lawful flight.

6.0

Regulatory Context

The platform does not directly implement regulatory compliance. The operator is responsible. Design choices that touch regulatory scope:

  • Drone weight. The airframe plus battery plus payload is designed to remain below 250 g all-up. This keeps recreational operation within the lower-burden recreational exemption in the FAA Part 107 framework in the United States. Commercial or autonomous-beyond-line-of-sight operation would require Part 107 certification and possibly LAANC airspace authorization.
  • FCC radio certification. All radio-emitting subsystems (Wi-Fi, ELRS, telemetry) use off-the-shelf, pre-certified modules (XIAO ESP32-S3, BETAFPV LiteRadio, etc.). The platform does not introduce new RF emissions that would require certification.
  • Privacy / video capture. The drone captures aerial imagery of the operator's own yard for mapping purposes. No cloud upload, no persistence beyond the mission. Capture of neighboring property is avoided by flight-path design (center-of-yard, bounded altitude). An explicit review of FAA §107 and state privacy statutes is an open action for iteration 2.
  • UL / CE marking. Not in scope for iteration 1. Productization (iteration 4+) will require safety certification including motor/battery/blade assessments.
7.0

Assumptions

The system assumes the following about the environment and the operator:

  • A1. The operator is a technically engaged adult who has read the README and understands the physical e-stop location.
  • A2. The yard is on private property under the operator's control. No third-party consent is required for operation.
  • A3. The operator's mobile device and the rover are on the same Wi-Fi network. Cross-network operation (e.g., cellular to home) is not supported.
  • A4. GPS, if enabled, has sky view. Iteration 1 makes GPS optional; visual SLAM is the primary localization source.
  • A5. Children and pets are known to the operator and supervised; the detection system is a safety backstop, not a substitute for supervision.
  • A6. The operator checks battery state before deploying. Iteration 1 has no automatic return-to-charge.
8.0

Open Issues

  1. OPEN-01 — Weather sensing. Iteration 1 relies on operator judgment for wind/rain. Evaluate adding a local weather-API poll or a barometric/humidity sensor in iteration 2.
  2. OPEN-02 — Home-network resilience. If the operator's Wi-Fi is down, the mobile app cannot issue commands. The rover's fail-safe is to stop — but this also means remote e-stop is unavailable. Consider a direct-to-rover Wi-Fi AP mode.
  3. OPEN-03 — Privacy review. Explicit review against state statutes regarding aerial imagery of neighboring property. Currently mitigated by flight-path design only.
  4. OPEN-04 — The "children and pets supervised" assumption (A5) is load-bearing for the safety argument. Needs a more rigorous hazard analysis in iteration 3 as blade is introduced.
