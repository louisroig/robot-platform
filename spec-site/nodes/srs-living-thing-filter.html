living_thing_filter

Bifurcates the raw YOLO detection stream into living things (for safety) and non-living things (for navigation).

Last Updated

1.0

Purpose

Subscribes to /perception/detections and splits it into /perception/people (safety-critical) and /perception/obstacles (navigation). The split is by class name against a configured whitelist.

2.0

Scope

In: class-based routing, forward-only filtering (no tracking, no confidence reweighting). Out: any temporal smoothing; size- or motion-based classification.

3.0

Interfaces

Subscriber: /perception/detections. Publishers: /perception/people, /perception/obstacles, /diagnostics.

4.0

Functional

  • SRS-PER-002-F01
    For each incoming DetectionArray, partition detections into the two output topics by matching class against the living-thing whitelist.
    verified by: TEST-PER-021
  • SRS-PER-002-F02
    Preserve the original detection fields (confidence, position, distance) unchanged.
    verified by: TEST-PER-022
  • SRS-PER-002-F03
    Publish both topics in the same callback, within the same cycle as the upstream arrival.
    verified by: TEST-PER-010
5.0

Performance

Added latency ≤ 2 ms; CPU usage negligible (< 1%).

6.0

Safety

  • SRS-PER-002-S01
    The whitelist shall err on the side of inclusion; any class that could plausibly be a living thing shall be routed to /perception/people.
    maps to: SR-002
7.0

Dependencies

rclpy, platform_msgs.

8.0

Configuration

Single parameter: living_things — list of class strings. Default: [person, dog, cat, bird, horse, sheep, cow, bear]. Override via application config.

9.0

Failure Modes

Upstream silence → filter silent. Own crash → systemd restart; during gap, safety_monitor staleness triggers ESTOP on /perception/people.

10.0

Verification

TEST-PER-010 (person latency), TEST-PER-011 (whitelist override), TEST-PER-012 (deadline diagnostic), TEST-PER-020 (obstacle pass-through), TEST-PER-021 (partition completeness), TEST-PER-022 (field preservation).

11.0
