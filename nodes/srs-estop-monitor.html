estop_monitor

GPIO-level observer of the physical e-stop button. The software echo of a hardware safety signal.

Document ID
SRS-HAL-003

1.0

Purpose

Reads the physical e-stop button via GPIO at 50 Hz, debounces, and publishes on /hal/estop_pressed. The node is an observer only — the hardware path (NC switch in series with motor power) already enforces the stop. This node informs the software stack so the safety monitor can propagate state.

2.0

Scope

In: GPIO polling, debounce, publication. Out: any actuation; multiple-input aggregation (the rover has exactly one physical e-stop).

3.0

Interfaces

Publishers: /hal/estop_pressed, /diagnostics. No subscribers.

4.0

Functional Requirements

  • SRS-HAL-003-F01
    Sample GPIO BCM 17 at 50 Hz, inverted (NC fail-safe: open = pressed).
    verified by: TEST-HAL-040
  • SRS-HAL-003-F02
    Debounce: require 3 consecutive matching samples before changing reported state.
    verified by: TEST-HAL-043
  • SRS-HAL-003-F03
    Publish at keepalive rate 20 Hz in addition to on-change.
    verified by: TEST-HAL-042
5.0

Performance

Button-press-to-topic latency ≤ 100 ms (p95). CPU usage < 2%.

6.0

Safety

  • SRS-HAL-003-S01
    NC-wire-break shall fail-safe to "pressed" (open circuit → GPIO high → inverted → TRUE).
    verified by: TEST-HAL-041
7.0

Dependencies

  • rclpy, std_msgs.
  • libgpiod (or gpiozero with libgpiod backend).
8.0

Configuration

ParameterDefault

9.0

Failure Modes

  • GPIO read exception → retry once → halt publication → diagnostic.
  • Chip not accessible on startup → node refuses to start.
10.0

Verification

TEST-HAL-040 (latency), TEST-HAL-041 (fail-safe wire break), TEST-HAL-042 (keepalive), TEST-HAL-043 (debounce).
