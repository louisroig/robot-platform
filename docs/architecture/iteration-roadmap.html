Iteration Roadmap

Four iterations, roughly 18 months, from prove-the-platform to productization prep. What is in scope for each iteration, and — equally important — what is explicitly out.

1.0

Purpose

This document defines the iteration structure of the project as of April 2026. Its purpose is twofold: to lay out the long arc so that iteration 1 decisions can be made with iteration 3 in mind, and to establish scope discipline that keeps iteration 1 from drifting into iteration 2's territory.

Every "this is iteration 2" or "iteration 3+" reference elsewhere in the corpus ultimately resolves to the definitions in this document.

2.0

Iteration 1 — prove the platform

Timeframe: 6 months, targeting an M6 field test demo.

What it proves:

  • The platform-first architecture survives contact with real hardware.
  • A drone can autonomously map a yard well enough to draw a boundary on.
  • A tracked rover can be teleoperated within that boundary with end-to-end safety.
  • The safety architecture works: e-stop, tilt, person-detect, geofence all cut motion as specified.

In scope:

  • All 14 custom ROS nodes enumerated in the node catalog.
  • All 21 ICDs in the interface catalog.
  • Mobile app with: "Map My Yard" button, polygon-draw UI, teleop joystick, live telemetry, soft e-stop.
  • Full test suite: unit, integration, and the M6 field test protocol.
  • A single working demo path (phases A–F in ARCH-003).

Out of scope — explicitly:

  • Autonomous navigation. Nav2 is integrated as a dependency only; the rover is teleop-only.
  • Coverage planning. Not built.
  • Multiple mission queuing.
  • Cloud services. Strictly local.
  • The blade / mower tool. The rover has no cutting implement in iteration 1.
  • Automatic charging / docking.
  • Safety MCU (remains software-only; hardware e-stop + software safety monitor).
  • Compliance engineering. Part 107, UL, CE all future.

Pass criterion: The M6 field test procedure runs to completion on five consecutive trial days.

3.0

Iteration 2 — autonomy

Timeframe: 3–6 months post-M6.

What it adds:

  • Nav2 waypoint navigation. The rover can be sent to a pose and will get there, respecting obstacles.
  • Coverage planner. Given a zone polygon, plan a coverage path. First-generation: simple boustrophedon.
  • Obstacle avoidance. Populate the costmap with /perception/obstacles; local planner respects it.
  • Behavior tree. Orchestrates coverage missions with pause/resume, battery-aware interruption.
  • Autonomous dock approach. Fiducial-based approach to a known dock position.
  • GPS / optional RTK. Upgrade localization fidelity to support larger yards.

Architectural changes: Layer 4 gains real content. Layer 5 gains queuing, scheduling. Platform-vs-application boundary is exercised for the first time by coverage logic.

Still out: The physical blade. Commercial certification prep.

4.0

Iteration 3 — mower chassis

Timeframe: 3–6 months post-iteration-2.

What it adds:

  • Actual mower chassis. Move off the tracked iteration-1 rover onto a purpose-built wheeled platform.
  • Blade tool. Add a cutting deck with its own motor, safety interlocks, and tool_controller node (mower-layer).
  • Safety MCU. The independent hardware safety board. Takes over blade power, enforces heartbeat + tilt + lift interlocks.
  • Blade-specific state machine. Tool engage/disengage states; lift-interrupt behavior.
  • Lift sensor. Detects if the mower has been picked up.
  • Surface classification. Distinguishes grass / pavement / soil; platform exposes classes, mower reacts.

Architectural changes: First real test of the platform/application boundary under pressure. The tool_controller, /hal/tool_cmd_safe, and blade-specific safety logic all live in applications/mower/. The platform gains generic tool concepts in platform_safety without naming the blade.

5.0

Iteration 4+ — productization

Timeframe: Ongoing, indefinite.

  • Compliance. UL safety assessment, FCC confirmations, Part 107 for commercial drone operation.
  • Manufacturability. Bill of materials discipline, harness pre-fabrication, enclosure sealing (IP rating).
  • Fleet management. If we sell units, they need software update, diagnostics export, remote support.
  • Second application. A non-mower application that exercises the platform/application boundary seriously. Candidate: pool skimmer, garden inspection, snow clearance.
  • Hazard analysis & system safety. Formal HARA in line with ISO 13482 (for personal-care robots) or similar.

This iteration is intentionally vague. The iteration-3 demo clarifies which direction makes sense commercially.

6.0

Scope Discipline

Three rules:

  1. Do not build Nav2 integration in iteration 1. It's tempting. It will reward the work with a better demo. It will also push M6 out and force iteration-2 work to be redone.
  2. Do not design the iteration-3 safety MCU in iteration 1. The placeholders in safety_monitor (GPIO heartbeat, tool veto topic) are enough. Detailed design in iteration 2.
  3. Do not break the platform/application boundary "for just this one thing." See ARCH-005.
Scope creep triggers

If any of the following phrases come up in planning, stop and check against this roadmap: "…and also we could…", "…while we're there…", "…this would be easier if we just…", "…iteration 1 should also…".

8.0

Build Order

The milestone table in §7 says what each milestone delivers. This section says what must be working to deliver it — specifically, which nodes, ICDs, state machines, and test protocols need to be in place at each M-gate, listed in dependency order so the solo build has a forced sequence rather than a parallel wish-list.

A node or ICD listed under a milestone is expected to be at least first-pass functional — it honors the ICD's happy-path contract, publishes at the declared rate, and passes any cited unit test. Fault-branch polish and QoS tuning continue across later milestones.

Reading the tables

Items marked stub are placeholders that satisfy the interface contract without real behavior (e.g. safety_monitor at M1 passes cmd_vel_raw through to cmd_vel_safe unconditionally). Items marked (carried) were introduced at a prior milestone and must remain green. Upgraded means an existing stub is replaced with real behavior.

What this list is not

This is the minimum functional-by-milestone ordering, not a work breakdown. Integration shakedown, mobile-app polish, documentation, and field-test rehearsals sit outside this table and must be scheduled in addition to it.
