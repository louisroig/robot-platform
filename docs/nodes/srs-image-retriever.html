image_retriever

Pulls captured still images from the drone's XIAO ESP32-S3 over Wi-Fi and persists them to rover disk.

Document ID
SRS-MAP-002
Revision
0.2 · draft
Owner
platform_mapping
Language
Python (rclpy + requests)
Classification
Data transport
Last Updated
2026-04-19
1.0

Purpose

Provides a ROS 2 service /mapping/retrieve_latest_image that fetches the most recent capture from the XIAO's HTTP server and writes it to /var/lib/platform/images/<mission_id>/capture.jpg. Called by drone_mission_coordinator after landing.

2.0

Scope

In: HTTP fetch, retry with backoff, persistence, metadata sidecar. Out: image processing (georeferencer's job); multi-image mosaicking.

3.0

Interfaces

Service server: /mapping/retrieve_latest_image (platform_msgs/srv/RetrieveLatestImage per ICD-MAP-004). Client of the XIAO HTTP capture API (ICD-XIAO-001) — uses GET /latest.jpg in production. Publisher: /diagnostics.

4.0

Functional

  • SRS-MAP-002-F01
    Issue HTTP GET to http://xiao.local/latest.jpg; stream to disk at configured path.
    verified by: TEST-MAP-040
  • SRS-MAP-002-F02
    Retry up to 3× with exponential backoff (1 s, 2 s, 4 s) on transient failure.
    verified by: TEST-MAP-041
5.0

Performance

Download a 2 MB JPEG over 5 GHz Wi-Fi in < 5 s. Node CPU cost negligible.

6.0

Safety

Not safety-critical. Failure to retrieve image fails the mission cleanly; no motion consequences.

7.0

Dependencies

rclpy, requests, platform_msgs. XIAO HTTP API contract (ICD-XIAO-001); XIAO firmware (FW-XIAO) implements the server side.

8.0

Configuration

ParameterDefault
xiao_hostxiao.local
image_dir/var/lib/platform/images
http_timeout_s10
retry_attempts3
9.0

Failure Modes

XIAO unreachable (Wi-Fi signal, firmware crash): exhaust retries → service returns failure. Disk full: emit diagnostic; return failure.

10.0

Verification

TEST-MAP-040 (happy path), TEST-MAP-041 (retry on transient failure). Both also verify ICD-MAP-004 requirements at the service-call boundary.

11.0
