1.0

Purpose

The safety_monitor node is the single software authority over whether the rover may move and whether the tool (iteration 2+) may engage. It subscribes to every safety-relevant input the system produces — perception, IMU, e-stop, geofence, battery, upstream velocity commands — aggregates them, maintains a state machine, and publishes both the computed state and the gated motion command.

Its role is not to be clever. It is to be right. It fails safe at every opportunity, it treats absence of information as worst-case, and it never invents a way to allow motion that the architecture did not explicitly permit.

Safety architecture context

safety_monitor is one of three independent safety layers (see ARCH-002 §3). Its failure degrades safety but does not defeat it: the physical e-stop and (iteration 2+) safety MCU provide redundancy.

2.0

Scope

In scope. Real-time aggregation of safety inputs, gating of motion commands, publication of unified state, the /safety/reset service, startup self-test, and diagnostics.

Not in scope. Direct control of hardware (the node does not cut motor power — that is the e-stop relay's job). Waypoint planning. Mission orchestration. Anything iteration 2+ (blade interlock logic, safety MCU heartbeat) beyond placeholder code.

The node is one of the three safety layers and is the software-layer gate only. It runs as a lifecycle node under systemd with process restart enabled.

4.0

Functional Requirements

  • SRS-SAF-001-F01
    Aggregate every subscribed input into a single safety state at ≥ 10 Hz.
    verified by: TEST-SAF-010
  • SRS-SAF-001-F02
    Publish /hal/cmd_vel_safe passing through /hal/cmd_vel_raw only when state = OK or WARNING.
    verified by: TEST-SAF-011
  • SRS-SAF-001-F03
    Apply a controlled deceleration ramp (default 2.0 m/s²) when transitioning to ESTOP from non-zero velocity, except when triggered by tilt.
    verified by: TEST-SAF-018
  • SRS-SAF-001-F04
    Provide the /safety/reset service to clear latched conditions. The service shall verify the underlying condition is clear before transitioning.
    verified by: ICD-SAF-002
  • SRS-SAF-001-F05
    Refuse to publish state=OK until a startup self-test confirms all subscribed topics have been seen at least once and are within their rate windows.
    verified by: TEST-SAF-020
  • SRS-SAF-001-F06
    Include in /safety/state.reasons the list of currently-active triggers, human-readable.
    verified by: TEST-SAF-021
5.0

Performance Requirements

  • SRS-SAF-001-P01
    End-to-end latency from /perception/people detection to /hal/cmd_vel_safe zero shall be ≤ 50 ms (95th percentile).
    verified by: TEST-SAF-011
  • SRS-SAF-001-P02
    Latency from /hal/estop_pressed transition to /hal/cmd_vel_safe zero shall be ≤ 20 ms.
    verified by: TEST-SAF-012
  • SRS-SAF-001-P03
    CPU usage on Pi 5 shall be < 15% of one core under nominal load.
    verified by: TEST-SAF-025
  • SRS-SAF-001-P04
    Resident memory shall be < 100 MB.
    verified by: TEST-SAF-026
6.0

Safety Requirements

  • SRS-SAF-001-S01
    If any subscribed safety input is stale beyond its configured staleness window, the node shall treat that input as worst-case (e.g., stale /perception/people implies a person is present).
    maps to: SR-006
  • SRS-SAF-001-S02
    Latched ESTOP conditions shall require explicit service call to clear, even if the underlying trigger condition is no longer active.
    maps to: SR-007
  • SRS-SAF-001-S03
    Non-finite values (NaN, Inf) in any Twist field received on /hal/cmd_vel_raw shall result in published zero Twist and diagnostic warning.
    maps to: SR-010
  • SRS-SAF-001-S04
    Safety-critical callbacks (perception, IMU, e-stop) shall run in a reentrant callback group under a MultiThreadedExecutor, preventing blocking by non-critical work.
    maps to: SR-005
7.0

State Machine

Four-state machine: OK · WARNING · ESTOP · CRITICAL. Formal transition table, invariants, and timing in SM-SAF-001.

Summary of invariants (full list in the state-machine document):

  • INV-1. tool_permitted ⇒ state ∈ {OK, WARNING}.
  • INV-2. /hal/cmd_vel_safe = 0 when state = ESTOP (within decel ramp tolerance).
  • INV-3. /safety/state is published at ≥ 10 Hz under all conditions.
  • INV-4. Hardware heartbeat (iter 2+) pulses at 100 Hz with < 5 ms jitter.
  • INV-5. Latched ESTOPs require /safety/reset.
  • INV-6. /safety/reset succeeds only if underlying conditions are clear.
  • INV-7. Stale topic data is treated as worst-case.

Open Issues

  1. OPEN-01 — Iteration 2: integrate hardware heartbeat line to safety MCU. Scaffolding present; output not wired.
  2. OPEN-02 — Reason-code enumeration (vs. strings) for machine-reliable app handling. See ICD-SAF-001 OPEN-01.
  3. OPEN-03 — Formal verification of the state machine. Candidate: generate TLA+ model from spec. Deferred.
  4. OPEN-04 — Stale-input enforcement (per SRS-SAF-001-S01) lives in this node, not the teleop UI. Bench finding 2026-05-03 (M1 phone-joystick smoke test): the teleop_web page-side stale-link check resets on any inbound message and so does not latch when only /hal/imu/data goes silent (because /diagnostics from this node + motor_driver keeps arriving at 1 Hz). Harmless at M1 because this node is a pass-through stub, but at M2 the gate must enforce IMU staleness server-side via topic_staleness_ms per §6 / S01. The JS check should be reduced to UX-only (or removed) once this node enforces.
