/mapping/georeference

Compute the geographic corners of a captured aerial image from drone pose at shutter; publish the resulting AerialImageMeta and return it in the response.

1.0

Purpose

The /mapping/georeference service is the explicit trigger for the rover-side flat-ground georeferencing computation that turns a captured JPEG plus shutter-time drone pose into the four geographic corner coordinates of the image footprint. It is called by drone_mission_coordinator after /mapping/retrieve_latest_image has placed the JPEG on disk, in the GEOREF post-LANDED state of SM-DRN-001.

The resulting AerialImageMeta is also published on /mapping/latest_aerial_image (TRANSIENT_LOCAL) so late-joining subscribers — most importantly mission_api — pick it up without depending on this service call. The response carries the same struct so the coordinator does not have to round-trip through a topic to read what it just produced.

3.0

Interface Definition

Service type: platform_msgs/srv/Georeference.

# Request string image_path # absolute path to the captured JPEG on rover disk float64 capture_lat # drone latitude at shutter (degrees, WGS84) float64 capture_lon # drone longitude at shutter (degrees, WGS84) float32 capture_alt_m # altitude above takeoff at shutter (meters) geometry_msgs/Quaternion capture_attitude # body-from-world rotation at shutter builtin_interfaces/Time capture_time # FC timestamp at shutter --- # Response bool success string message # empty on success; reason on failure platform_msgs/AerialImageMeta meta # fully populated on success; default-init on failure

The meta field on success is identical (byte-for-byte) to the message published on /mapping/latest_aerial_image for this call.

4.0

Semantics

On invocation, georeferencer:

  1. Validates image_path exists and is a readable JPEG. Missing or unreadable → SUCCESS=FALSE.
  2. Validates capture_alt_m is finite. Negative or non-finite → SUCCESS=FALSE. (Out-of-range altitudes log a warning and still compute, per SRS-MAP-003 §9.)
  3. Computes the pinhole-camera footprint at the given altitude using configured camera_hfov_deg / camera_vfov_deg; rotates by capture_attitude; converts ground-frame corner offsets to lat/lon via pyproj from (capture_lat, capture_lon).
  4. Populates the AerialImageMeta with image_path, capture pose, four corner_lats / corner_lons (NE, SE, SW, NW), and capture_time.
  5. Publishes the meta on /mapping/latest_aerial_image.
  6. Returns SUCCESS=TRUE with a copy of the meta in the response.

The publish-and-respond order is: publish first, then respond. This ordering ensures that any subscriber on /mapping/latest_aerial_image sees the new meta no later than the caller does — the coordinator cannot race ahead of mission_api by reading the response and signaling completion before mission_api has the topic update.

Fault

FaultResponse
georeferencer downROS 2 service call fails (no server). Coordinator surfaces as mission failure with reason "georeferencer_unreachable".
image_path missing or unreadableSUCCESS=FALSE, message="image not found at <path>". No publication on the topic.
Non-finite or negative altitudeSUCCESS=FALSE, message="invalid capture_alt_m: <value>".
Quaternion non-finite or zero-normSUCCESS=FALSE, message="invalid capture_attitude".
pyproj transform raisesSUCCESS=FALSE, message="projection failed: <exc>"; diagnostic at WARN.

Failure does not publish on /mapping/latest_aerial_image: the topic carries only successful georeferences. The TRANSIENT_LOCAL last-value semantics mean a subscriber that joins late after a failure sees the previous successful georeference (if any), never a partial one.
