XIAO HTTP capture API

The wire-level contract between rover-side ROS nodes and the drone's XIAO ESP32-S3 image-capture HTTP server. Companion to FW-XIAO §4.

1.0

Purpose

Formalizes the HTTP interface that rover-side software (specifically image_retriever) talks to on the drone's XIAO ESP32-S3 to obtain captured aerial images. The XIAO firmware spec (FW-XIAO) describes the server side; this document is the consumer-facing contract: endpoints, response shapes, latency expectations, and what failure looks like to a Python requests caller.

The contract is normative for both sides. A change to either FW-XIAO §4 or this document requires a coordinated rev of the other.

3.0

Endpoints

3.1 GET /capture

Triggers a fresh capture. Stores the JPEG into the XIAO's rolling buffer and returns the same bytes inline.

FieldValue
MethodGET
Request bodynone
Request headersnone required; User-Agent recommended for log correlation
Success status200 OK
Success Content-Typeimage/jpeg
Success Content-Lengthset, equals JPEG byte count
Success bodyJPEG bytes; 1600×1200 (UXGA), quality 12 per FW-XIAO §4 (subject to camera config)
Error statuses503 (camera timeout), 500 (firmware fault), 429 (concurrent capture in flight, see §6)
Error bodyplaintext one-line reason; not machine-parsed
3.2 GET /latest.jpg

Returns the most-recently captured image without re-triggering the camera. This is the production endpoint for image_retriever — the coordinator triggers capture earlier (during the CAPTURING state, before landing) and uses /latest.jpg to fetch after landing.

FieldValue
MethodGET
Request bodynone
Success status200 OK
Success Content-Typeimage/jpeg
Success bodyJPEG bytes from the most recent successful capture
Error statuses404 (no capture has happened since last boot), 500 (firmware fault)

Both endpoints are idempotent in a HTTP sense (repeated GETs do not corrupt server state); semantically /capture is not idempotent — each call advances the rolling buffer to a new image. /latest.jpg is fully idempotent and safe to retry indefinitely.

4.0

Discovery & Transport

  • Discovery. The XIAO advertises itself as xiao-bridge.local via mDNS (FW-XIAO §3). Clients should resolve this name fresh per session — DNS caching beyond a few seconds risks pointing at stale Wi-Fi-leased IPs after a drone reboot. Static IP fallback is not provided; if mDNS is unreliable on the operator's network, fix the network rather than the spec.
  • Transport. HTTP/1.1 over TCP, port 80. No TLS — the link is a closed Wi-Fi network on a battery-powered platform; the cost of TLS termination on ESP32 outweighs the threat model for iter 1. Do not put credentials or PII over this interface.
  • Connection reuse. The XIAO accepts but does not require keep-alive; clients should close after each response. Concurrent connections to the HTTP server are not exercised by the production workload (image_retriever is the sole caller and serializes its calls); the firmware reserves the right to reject overlapping requests with 429 (see §6).
  • Compression. No Content-Encoding support. JPEG is already compressed; gzip would only add CPU load.

Timing

Recommended client timeout10 s per attempt (matches image_retriever's http_timeout_s default in SRS-MAP-002 §8)

Throughput depends heavily on RF conditions and may degrade severely past 30 m or behind obstructions; FW-XIAO OPEN-02 tracks the open question of yard-scale pipe characterization.

6.0

Concurrency

The firmware is single-threaded with respect to the camera. Behavior under overlapping HTTP requests:

  • Two concurrent GET /latest.jpg requests — both succeed; the buffer is read-only on this path.
  • A GET /capture while another /capture is in flight — the second request returns 429 Too Many Requests with body "capture_in_progress". The first request completes normally.
  • A GET /latest.jpg while a /capture is in flight — returns the previous capture (the one before the in-flight one). The buffer is not partially exposed; either the old image or the new image, never a mix.

Production callers are advised to serialize their own calls regardless. The 429 path exists to give bench operators a clean signal rather than mysterious timeouts.

8.0

Requirements

  • REQ-ICD-XIAO-001-01
    The XIAO HTTP server shall expose GET /capture and GET /latest.jpg on port 80 of xiao-bridge.local per §3.
    verified by: TEST-XIAO-001
  • REQ-ICD-XIAO-001-02
    A successful response shall include Content-Type: image/jpeg and a Content-Length matching the JPEG byte count, so a streaming client can detect truncation.
    verified by: TEST-XIAO-001
  • REQ-ICD-XIAO-001-03
    A second GET /capture issued while the first is still processing shall return 429, not block; the first request's success shall be unaffected.
    verified by: TEST-XIAO-002
  • REQ-ICD-XIAO-001-04
    After a XIAO reboot, GET /latest.jpg with no prior capture shall return 404 (not a stale pre-reboot image and not a malformed empty response).
    verified by: TEST-XIAO-003
  • REQ-ICD-XIAO-001-05
    The XIAO shall advertise its hostname over mDNS within 5 s of Wi-Fi association, so clients on the same LAN can resolve xiao-bridge.local without manual configuration.
    verified by: TEST-XIAO-004

10.0

Open Issues

  1. OPEN-01 — EXIF / sidecar metadata. The XIAO does not currently stamp the JPEG with capture time or pose; coordinator carries pose separately. If /mapping/georeference's pose-via-request pattern proves brittle, revisit and have the XIAO embed an EXIF block.
  2. OPEN-02 — Range / RF characterization is FW-XIAO OPEN-02; reflect findings here once measured (e.g. degrade timeout above some range).
  3. OPEN-03 — A small GET /healthz endpoint would let the coordinator do a pre-flight reachability check (does mDNS resolve, does the server answer) without burning a capture. Out of scope for v0.1.
