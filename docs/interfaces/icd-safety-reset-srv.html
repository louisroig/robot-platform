/safety/reset

Operator-initiated clear of latched safety states. Validates preconditions, never forces a clear.

Document ID
ICD-SAF-002
Revision
0.1 · draft

platform_safety

1.0

Purpose

Some ESTOP triggers require explicit operator acknowledgment before the system will resume. The physical e-stop button is the archetype: a person pressed it, and the system should not un-latch simply because they released it — the operator must confirm readiness to resume.

The /safety/reset service is the mechanism. It is called by the mobile app (via mission_api) when the operator taps "Resume." It performs precondition checks and, if all are met, clears the latch.

Never forces a clear

Calling /safety/reset does not override any active safety condition. If the button is still pressed, the tilt is still exceeded, or the person is still detected, the service returns success=FALSE with an explanation. The operator must physically resolve the condition first.

Interface Definition

Service type: platform_msgs/srv/ResetSafety.

# Request (empty) --- # Response bool success string reason # human-readable explanation on failure

Example response when button still pressed:

{ "success": false, "reason": "estop_button_pressed (clear the physical button before resetting)" }
4.0

Semantics

On invocation, safety_monitor inspects every active latched condition:

  1. If any latched condition remains active, response is FAILURE with the first-listed active condition in reason.
  2. If all latched conditions are clear but there are active auto-clearable warnings, the reset still succeeds (it handles latches only; auto-clearable conditions will clear themselves).
  3. If no latches and no warnings, state transitions to OK immediately; response is SUCCESS.

The call is idempotent: calling it when state is already OK returns SUCCESS immediately.

5.0

Timing

Service callback latency ≤ 100 ms. State change (if any) appears on /safety/state within one publication cycle (~100 ms).

6.0

Requirements

  • REQ-ICD-SAF-002-01
    The service shall return success=FALSE if any latched trigger condition is currently active; the reason string shall identify the condition.
    verified by: TEST-SAF-022
  • REQ-ICD-SAF-002-02
    On success, the state transition shall be reflected on /safety/state within one publication cycle (≤ 100 ms).
    verified by: TEST-SAF-023
  • REQ-ICD-SAF-002-03
    The service shall never force-clear an active trigger; the only effect of success is to clear the latch bit.
    verified by: TEST-SAF-024
