1.0

Purpose

The /hal/cmd_vel_safe topic is the single authorized channel by which velocity commands reach the ground unit's motor driver. It carries the output of the safety_monitor node — either a pass-through of the upstream commanded velocity (when the system is in a safe state) or a controlled zeroed value (when safety conditions are violated).

The topic exists to enforce a hard architectural invariant: no actuator motion occurs without explicit safety gating. Any subsystem capable of commanding the rover — teleoperation, autonomous waypoint navigation, return-to-dock behavior, emergency recovery — funnels through safety_monitor, which publishes to this topic. The motor driver subscribes only to this topic and has no other source of velocity commands.

Safety-Critical Interface

This interface is part of the Layer 2 safety architecture (see ARCH-002 §3). The hardware e-stop relay and (in iteration 2+) the independent safety MCU provide redundant paths. /hal/cmd_vel_safe is the software-layer gate; it is not the sole safety mechanism. Failure of this interface alone shall not defeat the overall safety architecture.

2.0

Participants

Exactly one publisher and exactly one subscriber are authorized for this topic. Additional subscribers for monitoring purposes are permitted; additional publishers are explicitly prohibited.

Publisher
safety_monitor
Package platform_safety SRS SRS-SAF-001 Responsibility Computes safety state from aggregated inputs. Publishes either the upstream velocity (gated passthrough) or a zeroed Twist with controlled deceleration ramp. Failure behavior On node crash, systemd restarts the process within 2 s. During the gap, the motor driver's command timeout (see §7) halts motors independently.
Subscriber
motor_driver
Package platform_hal SRS SRS-HAL-001 Responsibility Converts incoming Twist to per-side track velocities via skid-steer kinematics. Drives dual-PWM (RPWM/LPWM per track) via GPIO to the IBT-2 (BTS7960B) H-bridges, per HW-PI5-001. Failure behavior Implements cmd_vel_timeout_ms (default 500 ms). If no message received within timeout, commands zero velocity. Provides a second independent safety layer against communication loss.

Monitoring subscribers (non-normative)

The following nodes may subscribe for observation without participating in the control loop:

  • rosbag2 — records all commands for post-mission analysis.
  • telemetry_publisher — forwards current commanded velocity to the operator app.
  • system_monitor — tracks topic publication rate for health diagnostics; detects unauthorized additional publishers.
3.0

Message Definition

Message type: geometry_msgs/msg/Twist (ROS 2 standard).

Field Type Unit Semantics
linear.x float64 m/s Forward (+) / reverse (−) velocity of the rover. Range bounded by max_linear_vel parameter in motor driver (default ±0.7 m/s for iteration 1).
linear.y float64 m/s Lateral velocity. Unused for skid-steer platform. Receiving nodes shall ignore. Publishers shall send zero.
linear.z float64 m/s Vertical velocity. Unused. Publishers shall send zero.
angular.x float64 rad/s Roll rate. Unused. Publishers shall send zero.
angular.y float64 rad/s Pitch rate. Unused. Publishers shall send zero.
angular.z float64 rad/s Yaw rate. Counter-clockwise positive (right-hand rule, z-axis up). Range bounded by max_angular_vel parameter (default ±1.5 rad/s).
Rationale: why reuse geometry_msgs/Twist instead of a custom type?

A custom SafeTwist message type was considered and rejected. Reusing the standard Twist type maintains compatibility with off-the-shelf ROS 2 packages (Nav2, twist_mux, rviz visualization), simplifies adapter code, and keeps downstream reasoning identical to any other ROS 2 robot. Safety semantics are enforced by the topic name and the architectural rule that only safety_monitor may publish to it — not by carrying extra fields in the message.

4.0

Semantics

The value published on /hal/cmd_vel_safe at any instant is determined by the current safety state and the upstream commanded velocity on /hal/cmd_vel_raw.

4.1 Gated passthrough (states OK, WARNING, CRITICAL)

The message published on /hal/cmd_vel_safe is byte-for-byte identical to the most recent message received on /hal/cmd_vel_raw, subject to:

  • No upstream message received within raw_cmd_stale_ms (default 200 ms): publish zero Twist.
  • Upstream linear.x exceeds max_linear_vel: clip to maximum magnitude, preserve sign, log warning.
  • Upstream angular.z exceeds max_angular_vel: clip, preserve sign, log warning.

4.2 Ramped halt (state ESTOP)

Upon entry to ESTOP, safety_monitor begins publishing a decelerating velocity profile instead of the upstream command. The profile satisfies:

// Let v₀ = velocity at t_estop (time of ESTOP entry) // Let t = current time, a = decel_rate_mps2 (default 2.0) linear.x(t) = sign(v₀) × max(0, |v₀| − a × (t − t_estop)) angular.z(t) = sign(ω₀) × max(0, |ω₀| − a × (t − t_estop)) // Once both reach zero, subsequent messages publish zero.

The ramp prevents mechanical shock to the chassis and reduces tip-over risk on inclines. Maximum ramp duration at default parameters is 0.7 / 2.0 = 0.35 seconds, bounded by the maximum commanded velocity.

Exception to the Ramp

If ESTOP is triggered by a tilt condition exceeding tilt_limit_deg, the ramp is bypassed and zero Twist is published immediately. Rationale: a tilting rover is already unstable; the ramp's purpose is to prevent destabilization, not to preserve it.

6.0

QoS Profile

The DDS Quality of Service profile for this topic is defined below. Both publisher and subscriber shall use compatible settings.

SettingValueRationale
ReliabilityRELIABLECommand loss is unacceptable; commands must arrive.
DurabilityVOLATILELate-joining subscribers do not need historical commands.
HistoryKEEP_LASTOnly the most recent command is relevant.
Depth1Deliberately shallow — prevents queue buildup during transient slowdowns. Older commands would be stale and dangerous to apply.
Deadline50 msSubscriber receives DEADLINE_MISSED event if publication exceeds this interval, triggering safe-halt.
Lifespan100 msMessages older than 100 ms are discarded by the subscriber before delivery.
QoS Compatibility

ROS 2 QoS compatibility rules require the subscriber's requested profile to be equal or weaker than the publisher's offered profile. The settings above are symmetric; both participants use identical profiles. Monitoring subscribers (e.g., rosbag2) may use weaker profiles (e.g., BEST_EFFORT) without disrupting the primary subscription.

Requirements

Requirements levied on this interface. All are verifiable via the test cases listed in §9.

  • REQ-ICD-002-01
    The publisher shall emit messages on /hal/cmd_vel_safe at 50 Hz ± 5 Hz during all non-fault operation.
    allocated to: safety_monitor · verified by: TEST-SAF-010
  • REQ-ICD-002-02
    The publisher shall emit a zero-velocity Twist within 50 ms of safety state entering ESTOP, following the decel ramp defined in §4.2.
    allocated to: safety_monitor · verified by: TEST-SAF-011
  • REQ-ICD-002-03
    The subscriber shall halt motor commands within 500 ms of the last received message.
    allocated to: motor_driver · verified by: TEST-HAL-009
  • REQ-ICD-002-04
    The subscriber shall reject messages containing non-finite values (NaN, ±Inf) in any Twist field and trigger safe-halt.
    allocated to: motor_driver · verified by: TEST-HAL-011
  • REQ-ICD-002-05
    Only safety_monitor shall publish on /hal/cmd_vel_safe. The presence of additional publishers shall be detected at runtime and treated as a safety fault.
    allocated to: system_monitor · verified by: TEST-SAF-016
  • REQ-ICD-002-06
    The QoS profile of both publisher and subscriber shall match §6. Deviations shall be caught by TEST-QOS-003 at startup.
    allocated to: safety_monitor, motor_driver · verified by: TEST-QOS-003
  • REQ-ICD-002-07
    Tilt-triggered ESTOP shall bypass the decel ramp and publish zero Twist immediately.
    allocated to: safety_monitor · verified by: TEST-SAF-018

10.0

Open Issues

  1. OPEN-01 — Decel rate (2.0 m/s²) is a first-pass estimate. Requires field validation on actual tracked chassis with varied surface conditions. May need per-surface calibration.
  2. OPEN-02 — Unclear whether to include a header timestamp field. Standard Twist lacks one. Consider extending to TwistStamped if latency diagnostics become necessary. Evaluate at iteration 1 retrospective.
  3. OPEN-03 — ROS 2 does not natively prevent multiple publishers on a topic. REQ-ICD-002-05 relies on runtime detection rather than architectural prevention. Consider DDS partitions or secure ROS 2 (SROS2) for hard enforcement if the safety case requires it.
  4. OPEN-04 — Iteration 2 (with blade) introduces /hal/tool_cmd_safe as a peer to this topic. The interaction between velocity gating and tool gating needs definition. Some safety states may permit motion while prohibiting tool engagement (e.g., return-to-dock on low battery).
