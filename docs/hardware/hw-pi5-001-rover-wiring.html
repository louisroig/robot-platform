1.0

Purpose

This document is the single source of truth for the Pi 5 rover's physical wiring. It freezes the BCM pin assignments used by the motor_driver, estop_monitor, and imu_driver SRSes, documents the per-device wiring, and provides a bring-up procedure. It closes OPEN-03 of SRS-HAL-001.

Downstream: the ROS parameter YAML defaults, the deploy/udev/ rules, and the electrical schematic all derive their pin numbers from this document. When pins change here, they change everywhere — do not fork.

3.0

GPIO & PWM Allocation

3.1 · Pi 5 header pins in use

All pins are Pi 5 40-pin header, BCM numbering. Header pin numbers given for physical reference.

FunctionBCM GPIOHeader pinDirectionNotes

GPIO 12, 13, 18, 19 (the Pi 5 hardware-PWM-capable pads) are intentionally unused in this revision — see §3.3. GPIO 14/15 are intentionally left unused so the UART console remains available for debugging. The CSI/DSI lanes, SPI flash pins, and PoE pads are untouched.

3.2 · PCA9685 channel allocation

The four IBT-2 PWM inputs are driven by a PCA9685 16-channel I²C PWM controller, not by Pi GPIOs. Channel numbers below are the indices used by the motor_driver ROS parameters (rpwm_left_pin, etc.) — the parameter name retains the legacy _pin suffix; for the pca9685 backend the value is a PCA9685 channel index in [0, 15], not a BCM GPIO.

FunctionPCA9685 channelConnects to
Left track — RPWM (forward)0Left IBT-2 RPWM input
Left track — LPWM (reverse)1Left IBT-2 LPWM input
Right track — RPWM (forward)2Right IBT-2 RPWM input
Right track — LPWM (reverse)3Right IBT-2 LPWM input

PCA9685 I²C bus 1, default address 0x40 (no jumpers shorted). Channels 4–15 are unused and may be reserved for M2+ peripherals (e.g. servo-actuated mower deck, status LED). Frequency is global across all 16 channels and is set to 1 kHz in the default motor_driver params (within the IBT-2's 0.5–25 kHz tolerance and the PCA9685's 24–1526 Hz hardware range).

3.3 · Why off-board PWM

RP1 PWM is unusable on this stack

This revision moves PWM generation off the Pi entirely. On the rover's Pi 5 / Ubuntu 24.04 / RP1 stack, two on-Pi PWM paths were tried and both failed:

  • lgpio.tx_pwm returns a non-negative wave-id (looks like success) but the pad stays at 0 V. Discovered 2026-05-02 during M1 motor bringup.
  • sysfs hardware PWM via the pwm-2chan overlay exposes npwm=4 on /sys/class/pwmchip0, but enabling any channel returns EINVAL — the rpi-pwm driver cannot resolve its clock rate on RP1.

Static gpio_write via lgpio works (a Python bit-bang loop at ~1 kHz successfully spun both motors on the bench), but it is too jittery and CPU-hungry to ship inside the motor_driver ROS node. The PCA9685 sidesteps the entire RP1 PWM stack: PWM is generated by a dedicated chip clocked off its own 25 MHz internal oscillator, and the Pi just sends I²C register writes when the duty changes.

4.0

Motor Drive · IBT-2

Two IBT-2 boards, one per track. Each IBT-2 has two BTS7960B chips internally wired as a single H-bridge; one IBT-2 = one motor. Driven in dual-PWM mode (not simple PWM+DIR mode) to get proper bidirectional control with regenerative braking at zero.

Address selection: leave A0–A5 jumpers all unshorted for the default 0x40. If a second PCA9685 is added later, short A0 to bias the new board to 0x41.

4.4 · Dual-PWM signalling convention

The software drives each track with a signed duty value in [−1.0, 1.0]:

Commanded valueRPWM outputLPWM outputMotor behavior
0.00 %0 %Coast / regen brake
+v (0<v≤1)v × 100 %0 %Forward at duty v
−v0 %v × 100 %Reverse at duty v

Driving both RPWM and LPWM non-zero simultaneously would short-circuit the motor through the H-bridge and risks board damage. Every backend in gpio_backend.py (Pca9685Backend, LgpioBackend, MockGpioBackend) enforces the mutual-exclusion invariant by construction — the opposite channel is zeroed before the new direction is activated.

4.5 · Decoupling

Install a 1000 µF / 25 V+ electrolytic capacitor across each IBT-2's B+ and B− terminals, as close to the terminal block as practical. This absorbs back-EMF transients from aggressive braking and stabilizes the motor supply rail during stall events.

5.0

E-stop

The e-stop must perform two independent functions simultaneously: cut motor power in hardware, and signal the ROS stack. This requires a DPST-NC switch — two normally-closed contacts that open together when the mushroom is pressed.

5.1 · Switch specification

ParameterMinimumNotes
Form factor22 mm mushroom, twist-releasePanel-mount, threaded bezel
Pole configurationDPST-NCBoth contacts normally closed
Motor-side rating≥ 10 A @ 24 VDCMargin over 2× 5.5 A stall worst case
Signal-side ratingAny (logic level)Carries < 1 mA at 3.3 V

5.2 · Wiring

Pole 1 · motor power cut:

Wire the NC contact in series with the battery positive lead, between battery+ and the B+ terminals of both IBT-2 boards. Pressing the mushroom opens this contact and removes motor power at the hardware level, independent of any software.

Pole 2 · signal to Pi:

Wire the NC contact between Pi 3V3 (header pin 17) and GPIO 17 (header pin 11). Enable the internal pull-down on GPIO 17 in the estop_monitor node.

Switch statePole 1Pole 2GPIO 17Interpretation
Normalclosed · motor power onclosed · 3V3 to GPIO 17HIGHSafe to operate
Pressedopen · motor power cutopen · pull-down winsLOWEmergency
Wire broken—open · pull-down winsLOWEmergency (fail-safe)
Failsafe polarity

The "HIGH = normal, LOW = emergency" convention is deliberate. A broken wire, a disconnected switch, a boot-before-cable-plugged sequence, or a power-glitched input all default to the safe interpretation: stop. The inverse convention would allow any wiring fault to silently defeat the e-stop.

6.0

IMU · Qwiic

The ISM330DHCX breakout connects via a SparkFun Qwiic HAT or SHIM, which routes Pi GPIO 2 (SDA) and GPIO 3 (SCL) plus 3V3 and GND to a JST-SH 4-pin Qwiic connector. A single Qwiic cable from HAT to breakout carries all four signals. No individual pin wiring required.

The PCA9685 (§3.2) shares this same I²C bus 1 — daisy-chain the second Qwiic port on the IMU breakout (or chain through the HAT) to the PCA9685's STEMMA QT input. Addresses do not collide (IMU 0x6A/0x6B, PCA9685 0x40).

7.0

Power & Grounding

7.1 · Rail separation

Three independent power rails on the rover:

  • Motor bus · 12 V. Direct from battery, through e-stop pole 1, to both IBT-2 B+ terminals. High current (up to 11 A peak), electrically noisy.
  • Pi 5 V. Derived from the 12 V bus through a dedicated buck converter (minimum 3 A rated; Pi 5 draws up to 2.5 A at full load). Must be filtered; share only ground with the motor bus, not the 12 V rail directly.
  • Logic 3.3 V. Generated by the Pi itself, exposed on header pins 1 and 17. Powers IBT-2 logic side, e-stop signal pole, and the Qwiic bus (IMU and PCA9685). The PCA9685's V+ servo rail is unused — leave it open; the IBT-2 inputs are driven by the chip's digital PWM pins.

7.2 · Star ground

A single physical tie point — a screw terminal block or ring-lug junction — collects:

  • Battery negative terminal
  • Both IBT-2 B− terminals
  • Both IBT-2 logic GND terminals
  • PCA9685 GND (typically through the Qwiic chain to the Pi, but a dedicated star return is preferred for switching-noise isolation)
  • Pi 5 V buck converter output negative
  • Pi header GND (any of pins 6, 9, 14, 20, 25, 30, 34, 39 — one is sufficient)

Do not daisy-chain grounds; each component gets its own run back to the star. Daisy-chained grounds introduce resistive voltage offsets that cause spurious GPIO reads and I²C errors under load.

7.3 · Battery

12 V nominal, minimum 10 A continuous discharge, minimum 5 Ah for a 30-minute demo run. A 3S LiPo (11.1 V nominal, 12.6 V full charge) at 3000–5000 mAh with ≥10 C rating meets all three. A 12 V 7 Ah SLA works but adds weight and charges slowly.

The motor supply range for XR25-370 is nominal 12 V; 3S LiPo at full charge (12.6 V) is within tolerance. 4S (14.8 V) exceeds the motor rating and risks premature brush wear; not recommended.

8.0

Bring-up Checklist

Work through these in order. Do not skip. Each step is a gate: if it fails, stop and fix before proceeding.

8.1 · Pre-power

  1. Confirm battery is disconnected and e-stop is pressed (open).
  2. Visually verify all four motor-side wires are secure in screw terminals (no loose strands bridging terminals).
  3. Visually verify star ground has every connection listed in §7.2 (battery −, 2× IBT-2 B−, 2× IBT-2 GND, PCA9685 GND, Pi GND, buck converter −).
  4. Check continuity from star ground to each end-point with a multimeter; < 0.1 Ω expected.
  5. Verify no continuity from battery + to star ground (checks for wiring shorts).

8.2 · Logic-only power-on

  1. Power Pi 5 from USB-C wall supply (not from the buck converter yet). Battery stays disconnected.
  2. Boot into Ubuntu Server 24.04. Log in.
  3. Enable I²C: sudo raspi-config nonint do_i2c 0. Reboot if prompted.
  4. Run i2cdetect -y 1. Expect to see 0x40 (PCA9685) and 0x6A or 0x6B (ISM330DHCX). If either is missing, check the Qwiic cable seat for that device.
  5. Run gpioinfo. Verify GPIO 17 is not claimed by any existing service. (GPIO 12/13/18/19 are no longer used in this revision — see §3.)

8.3 · Motor bench test (tracks off ground)

  1. Rover on blocks, tracks free to spin without contact.
  2. Release e-stop (twist to un-latch).
  3. Connect battery.
  4. Verify motor supply by checking voltage at each IBT-2 B+ / B−: should read battery voltage (~12 V).
  5. From ROS 2: ros2 launch platform_hal platform_hal.launch.py
  6. Publish a low forward command: ros2 topic pub -r 50 /hal/cmd_vel_raw geometry_msgs/Twist "{linear: {x: 0.1}}"
  7. Both tracks should spin forward. Note any unexpected behavior before continuing.
  8. Press e-stop. Both tracks must stop within < 100 ms of press. If they do not, disconnect battery immediately and recheck pole 1 wiring.

8.4 · Direction-polarity check

  1. With tracks still off the ground, command linear.x: 0.1 and confirm both tracks spin in the same direction (both wind the rover forward, not against each other).
  2. If the left track spins opposite to the right: swap M+ and M− on the left IBT-2 only. Do not invert signs in software.
  3. Command angular.z: 0.5 and confirm the rover would skid-turn counter-clockwise (left track backward, right track forward).
  4. If turn direction is inverted, swap M+ and M− on both IBT-2 boards. Then redo the previous step.

8.5 · On-ground test

  1. Rover on hard flat ground. Clear space ≥ 3 m in all directions.
  2. Operator has e-stop within immediate reach.
  3. Publish linear.x: 0.2 for 2 seconds. Rover moves forward ~0.4 m.
  4. Measure actual distance traveled. Divide by 2 s → actual velocity. If significantly < 0.2 m/s, calibration adjustment needed (see SRS-HAL-001 OPEN-01).
  5. Ramp to linear.x: 0.7 (spec max) and measure. This is the empirical max linear velocity; update max_linear_vel parameter to match.
