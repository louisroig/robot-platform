STATE MACHINE · MISSION ORCHESTRATION

mission_executor state machine

Top-level behavioral state: what the rover is doing right now. Iteration 1 uses only a subset (IDLE · MAPPING · TELEOP); MOWING / PAUSED / RETURNING are iteration-2 placeholders.

1.0

Purpose

Captures the rover's activity state as observed by the operator. This is distinct from the safety-monitor state machine (which covers whether motion is permitted). The two run in parallel; the safety state constrains what this machine can do.

In iteration 1 there is no single "mission_executor" node — mission state is derived by mission_api and the drone coordinator. Iteration 2 introduces a formal behavior tree owning this state.

5.0

Iteration 1 Simplification

In iteration 1, mission state is derived rather than explicitly held by a single node. The mobile app infers state by combining:

  • The action server's reported mission state → MAPPING.
  • Recency of /hal/cmd_vel_teleop messages → TELEOP.
  • Otherwise → IDLE.

The /rover/state endpoint returns a mission_state string derived this way. When a formal mission_executor node is introduced in iteration 2, it will own this state and publish it on a dedicated topic; the derivation logic will move from the client to that node.
