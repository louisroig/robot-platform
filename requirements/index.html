Requirement taxonomy

Requirements are organized by type, not by component. Each requirement has a stable ID of the form <prefix>-<num>, an allocation (which node owns it), a verification method (how it is proven), and a trace to one or more test cases.

PrefixTypeMeaning
FR-xxxFunctionalWhat the system does — capabilities observable to the user.
SR-xxxSafetyHazard controls, fail-safe behaviors, independence requirements.
PR-xxxPerformanceRates, latencies, throughput, coverage, battery life.
IR-xxxInterfaceObligations on the contracts between components (complement to ICDs).
REQ-ICD-xxxInterface-localRequirements scoped to a single ICD, numbered under its document ID.
